# Extractor precision/recall — labeling workflow

Issue: arunsanna/mcp-bom#16

## Goal

Establish ground-truth precision/recall and Cohen's kappa for the static
extractor against a stratified sample of MCP servers. Targets:

- precision >= 0.80 per category
- recall >= 0.75 per category
- Cohen's kappa >= 0.70 overall

## Files in this directory

| File                    | Purpose                                                                                                                                         |
| ----------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------- |
| `select_sample.py`      | Picks N stratified servers from `corpus/manifest.json`, filtered to those with real local archives. Reproducible via `--seed`.                  |
| `generate_worksheet.py` | For each sampled server, unzips it, runs the extractor, writes `predictions.json` (extractor output) and `labels.template.json` (blank labels). |
| `score_metrics.py`      | Reads `predictions.json` + `labels.json`, computes per-category precision/recall + Cohen's kappa. Emits `metrics.md`.                           |
| `sample.json`           | The sampled server list (regenerate with `select_sample.py`).                                                                                   |
| `predictions.json`      | Latest extractor predictions per server.                                                                                                        |
| `labels.template.json`  | Empty labels file — copy to `labels.json` and fill in.                                                                                          |
| `labels.json`           | Your ground-truth labels (gitignored if you want; commit if part of paper).                                                                     |
| `metrics.md`            | Generated report.                                                                                                                               |

## Step-by-step

```bash
# 1. Select stratified sample (reproducible; default n=30, seed=1729)
. extractor/.venv/bin/activate
python validation/precision_recall/select_sample.py

# 2. Run extractor over the sample, generate the labels template
python validation/precision_recall/generate_worksheet.py

# 3. Copy the template and label
cp validation/precision_recall/labels.template.json \
   validation/precision_recall/labels.json
# ... open labels.json, set each per-category label to true/false ...

# 4. Score
python validation/precision_recall/score_metrics.py
# -> writes metrics.md
```

## How to label

For each server in `labels.json`:

1. Open the `repo_url` in a browser.
2. Decide for each category whether the server _exposes that capability to the
   LLM_ (not whether the source merely contains a relevant import).
3. Set `labels.<category>` to `true` (capability is exposed) or `false` (not exposed).
4. Add a one-line `notes` if the case was tricky (helps later auditors).

A capability is **exposed** when the server registers a tool / resource /
prompt that allows the LLM to invoke it. Internal implementation details
(e.g., the server reads its own config from disk) are NOT exposed capabilities.

### Per-category quick rules

- **filesystem**: Tool that takes a `path`/`filename`/`directory` argument and
  reads, writes, or deletes. Server reading its own config from disk → no.
- **shell**: Tool that runs a command, script, or arbitrary code. Server using
  subprocess for an internal helper → no.
- **egress**: Tool that fetches a URL or queries a remote datastore on behalf
  of the LLM. Server hitting a single fixed API to wrap it → only if that API
  is the wrapped capability.
- **ingress**: Server binds a network port for incoming requests (HTTP/SSE/WS).
  stdio-only server → no.
- **secrets**: Tool that lets the LLM read env vars / secret store. Server
  reading `os.environ.get("LITERAL_KEY")` for its own config → no (Tier 1);
  tool that reads `os.environ.get(varname)` where `varname` is a parameter →
  yes (Tier 2).
- **delegation**: Server itself is an MCP client of _other_ MCP servers.
  Comments mentioning "Client" → no.
- **impersonation**: Server can post on behalf of the user (Slack, email,
  GitHub PR, etc.). Server that only reads → no.
- **data_sensitivity**: Server's domain is PII, PHI, financial, or location
  data (Notion docs → likely yes; calculator → no).

## Reproducibility

The sample is deterministic for a given `--seed`. To re-do the sample with
different stratification, change `LANGUAGE_QUOTAS` in `select_sample.py`.

The `predictions.json` is regenerated by `generate_worksheet.py` and reflects
the current extractor; commit a snapshot if you want to compare against a
later version.
