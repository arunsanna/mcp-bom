EXTRACTOR PORT PARITY ANALYSIS
===============================
Date: 2026-05-02
Comparing: spike/extractor.py (reference) vs extractor/mcp_bom/extractor.py (port)
Corpus: spike v3 (n=27) vs production scan (n=298)

1. ARCHITECTURAL DIFFERENCES
----------------------------

1a. Database category dropped
   - Spike: 9 categories (filesystem, shell, egress, ingress, secrets, delegation, impersonation, data_sensitivity, database)
   - Port:  8 categories (database removed per taxonomy decision)

1b. Pattern structure
   - Spike: PATTERNS dict with lang-keyed sub-dicts (python, typescript) + schema patterns per category
     Spike pattern counts (python/typescript/schema):
       filesystem:       4/4/2    shell:           4/3/2    egress:      2/2/1
       ingress:          2/2/0    secrets:         2/2/2    delegation:  2/2/1
       impersonation:    2/2/2    data_sensitivity:1/1/2    database:    3/3/2
   - Port:  per-category modules with sub-typed pattern dicts (e.g. read/write/delete for filesystem)
     Port regex pattern counts:
       filesystem: 52    shell: 31    egress: 47    ingress: 28
       secrets: 15       delegation: 21    impersonation: 30    data_sensitivity: 33
   - Port has ~4x more patterns per category (more granular sub-types)

1c. Schema patterns DROPPED in port
   - Spike: 7 of 9 categories had "schema" patterns matching JSON schema property names (e.g. "path", "filename", "command")
   - Port: 0 of 8 categories have schema patterns
   - Impact: removes a source of false positives but also loses tool-schema-level detection

1d. Comment/string stripping ADDED in port
   - Port: strip_comments_and_strings() blanks out comments and string literals before regex matching
   - Spike: no stripping — comments and docstrings can trigger patterns
   - Impact: reduces false positives from commented-out code and documentation strings

1e. Tool-scope gating ADDED in port
   - Port: near_tool_reg() and python_tool_source() restrict detection to code within/near MCP tool registrations
   - Spike: scans all source files with no scope restriction
   - Impact: most significant architectural change — see Section 2

1f. Tier system ADDED in port
   - Port: tier1 (import-level) + tier2 (usage-level) with configurable thresholds
   - Spike: single threshold per category

1g. AST augmentation in port
   - Port: uses Python ast module for import analysis, class detection
   - Spike: pure regex

5. ROOT CAUSES
--------------

5a. Tool-scope gating (PRIMARY)
    Port's near_tool_reg() and python_tool_source() only detect capabilities within/near MCP tool
    registration functions. Module-level imports, constructor initialization, and helper functions
    outside the tool boundary are invisible.
    
    Affected: filesystem (read/write in setup code), secrets (env vars read at module level)

5b. Schema patterns dropped (SECONDARY)
    Port dropped all schema patterns. This removes detection of capabilities declared in JSON
    schema (tool parameter names like "path", "command", "url").
    
    Affected: filesystem, shell, egress, secrets, delegation, impersonation, data_sensitivity

5c. Comment stripping (TERTIARY)
    strip_comments_and_strings() removes some true positives from docstrings that contain
    capability indicators (e.g., "reads from filesystem" in a docstring).
    
    Affected: minor impact across all categories

5d. Sample difference (CONFOUND)
    Spike n=27 vs port n=298 — different sample sizes explain some prevalence shift,
    but 65-68pp drops for filesystem/secrets cannot be explained by sample alone.


6. VERDICT
---------

Status: INCONCLUSIVE (with identified root causes)

The port extractor is NOT equivalent to the spike extractor. Systematic under-detection
in filesystem (-65.7pp) and secrets (-68.1pp) is explained by two architectural changes:

1. Tool-scope gating restricts detection to MCP-tool-registered code only
2. Schema patterns were dropped entirely

These are design choices, not bugs — the port is intentionally more conservative.
However, for the purposes of the MCP-BOM benchmark:

- If the benchmark aims to measure DEPLOYED capabilities (what the MCP server CAN do),
  the tool-scope approach is defensible but needs tool-boundary detection validation.
  
- If the benchmark aims to measure CODE-LEVEL risk (what the source code CONTAINS),
  the spike approach is more appropriate and the port is undercounting.

RECOMMENDATION:
1. Validate tool-scope boundary detection precision/recall (instrument validation phase)
2. Consider a "code-level" vs "tool-level" detection mode
3. Re-add schema patterns as a separate detection layer (not gated by tool scope)
4. Document the design choice and its impact in the paper's threats-to-validity section
